# 検証対象

**「意思のブロードキャスト装置」** — 事業者の情報/意思を、増殖するAIエンドポイント群に
構造化して自動配信する仕組み。SEO→AEOの延長線上にある次世代インフラ。

---

# 法則による検証（楽観/悲観の両面）

## LAW-004（摩擦崩壊）適用

**楽観シナリオ:**
- AIが「調べる側」の摩擦を崩壊 → 検索回数が爆増（ジェボンズ）
- 「見つけてもらう側」が新ボトルネック → AI配信装置に価値が集中

**悲観シナリオ:**
- 「見つけてもらう側」の摩擦は本当にボトルネックか？
- もし消費者がAIの回答を「とりあえず上位3つ」で満足するなら、見つけてもらう側の努力は報われない
- → **ボトルネックは「AIに載ること」ではなく「AIの回答の上位に載ること」**
- これはGoogleのSEOと同じ構造。SEOツール市場は$80B程度で巨大だが独占者はいない

**判定:** 価値は集中する（LAW-004 ✓）。ただし独占可能かは別問題（LAW-009へ）

---

## LAW-005（信用レイヤーの価値上昇）適用

**楽観シナリオ:**
- AI配信装置は「正確な情報を届ける」ための信用インフラ
- 信用レイヤーの価値が上がる → この装置の価値も上がる

**悲観シナリオ — ★ここが核心:**
> 「信用が摩擦の場合、思ったより広がらない/マーケットを取れない世界がある」

LAW-005の注意書きが明確に警告している:
> 「価値の集中は独占可能性を意味しない」
> 「信用レイヤーはCON-002×LAW-008の二重制約により、持続的な独占が構造的に不可能」

具体的に何が起きるか:
1. AI配信装置が「正確性を保証する」機能を持つと → **審判化する**
2. 審判化すると → LAW-008で構造的に腐る（売り手課金なら中立性が死ぬ）
3. 審判化しないと → ただのツール。信用レイヤーではない。差別化困難

**判定:** 信用レイヤーそのものを建てようとすると不可能領域に入る。
**回避策:** 信用を「判定する」のではなく「伝達する」に留める。
ルーターであって審判ではない位置を取る。

---

## LAW-009（ロックイン＝ルーティング）適用

**核心の問い:** AI配信装置はルーティング機能を持つか？

```
[事業者] → [AI配信装置] → [AIエンドポイント群]
                               ↓
                          [消費者]
```

**ルーティングの成立条件:**
- 事業者側: AI配信装置を経由しないとAIに載れない → ✓（各AIに個別対応は不可能）
- AI側: AI配信装置を経由しないと正確な情報を得られない → ★ここが問題

**悲観シナリオ:**
- AI各社は自分でクロールできる（GooglebotのAI版）
- AI配信装置を経由する「必要」がない
- → 片面のみの依存（事業者→装置）= LAW-009でロックイン不成立 = ツール止まり

**楽観シナリオ:**
- AI各社がクロールで得る情報は不正確/古い
- 事業者が「公式に」配信した情報の方が正確 → AI側も依存する
- → 両面依存が成立 → ルーティング成立

**歴史的アナロジー:**
- Google Search Console: 事業者がGoogleに構造化データを能動的に届ける
- → Googleはクロールできるが、構造化データを受けた方が精度が上がるので受け入れる
- → 結果的にGSCは「半両面」（依存度はGoogle >> 事業者 ではなく 事業者 >> Google）
- **GSCはルーティング装置ではなかった。Googleのルーティングに乗るためのツールだった**

**判定:** AI配信装置は**ルーターではなくツール**に終わる可能性が高い。
ルーティングを持つのはAI各社自身（ChatGPT/Gemini/Perplexity）。
AI配信装置はGSCの次世代版 = 「AIの上に乗るためのSaaSツール」。

## CON-002（信用＝責任の所在）適用

**問い:** AI配信装置が「この情報は正確です」と保証できるか？

- 保証する場合 → 責任を負う → コストが高い → スケールしにくい
- 保証しない場合 → ただのパイプ → 差別化困難

**判定:** パイプとして始め、実績データの蓄積で「このパイプを通った情報はAIに正確に引用される確率X%」という統計的信用を積む。保証はしないが結果で証明する。

---

## CON-009（インセンティブ超反応）適用

**問い:** 事業者にとって、AI配信装置を使うインセンティブは十分か？

- 今（2026）: AIからの流入が売上の5-10%程度 → まだ痛みが弱い
- 2028: AIからの流入が30%+ → 「AIに載っていない=死」→ 全員使う

**判定:** タイミング問題。今は early adopter のみ。マス化は2028-2030。

---

## CON-015（横断問題）適用

**問い:** 複数のAIエンドポイントに対応するには各社の協力が必要か？

- OpenAI/Google/Anthropic/Perplexity... 各社が独自フォーマットを持つ可能性
- ただし、schema.org/JSON-LDのような標準が既にある
- AI各社も正確な情報を欲しいので、標準フォーマットを受け入れるインセンティブあり

**判定:** CON-015は部分的。各AIが「公式データフィード」を受け入れる仕組みを作れば横断可能。
ただしAI各社が独自の「公式チャネル」を作ってしまう可能性（Google Merchant的な囲い込み）。

# 総合判定

| 法則 | 結果 | リスク |
|---|---|---|
| LAW-004 | ✓ 価値は集中する | 独占可能かは別問題 |
| LAW-005 | △ 信用レイヤーの罠 | 審判化すると不可能領域。伝達者に留まるべき |
| LAW-008 | ⚠ 審判化回避必須 | 「正確性保証」をしない設計が必要 |
| LAW-009 | ⚠ **最大リスク** | ルーティング不成立→ツール止まりの可能性。GSC的ポジション |
| CON-002 | △ 保証せず結果で証明 | 統計的信用を積む |
| CON-009 | △ タイミング問題 | 今は早い。2028-2030にマス化 |
| CON-015 | △ 部分的 | AI各社の囲い込みリスク |

## 最大の問い: 「ツール」で十分か？ それとも「ルーター」でないと死ぬか？

### ツール市場の場合（GSC的ポジション）:
- 市場規模: SEOツール市場が$80B → AEOツール市場も同等以上は可能
- しかし: 独占不能。Ahrefs/SEMrush/Moz的に複数プレイヤーが並立
- 利点: 確実に存在する市場。勝てなくても儲かる
- 欠点: 10x returnではない。SaaS的成長

### ルーター市場の場合（「AI版Google」）:
- ルーターになれれば: 10x〜100x return。勝者総取り
- しかし: AI各社自身がルーター。外部からルーター位置を奪うのは困難
- 唯一の窓: **AI各社が「正確な事業者情報を得る」ための統一フィードを事業者側から提供する構造** — つまり事業者側のルーターになる

### 第三の道: 「事業者→AI」のルーティング（片面だが強い依存）
- Google Search Console は片面（事業者→Google）だがGoogle Searchの一部として成立
- AI配信装置が「事業者→全AI」の標準パイプになれば、事業者側は乗り換えコスト高い
- AI側の依存は弱くてもよい（正確な情報が来るなら受け入れる）
- **事業者側のスイッチングコスト × 事業者数** でビジネスは成立する

# 結論: 「信用が摩擦」問題への回答

**「信用が摩擦の場合、広がらない/取れない世界」は確かに存在する。** 具体的には:

1. **審判として信用を判定しようとすると死ぬ**（LAW-008）
2. **信用レイヤーを独占しようとすると構造的に不可能**（LAW-009 ルーティング不成立）
3. **「保証」を売ると責任コストでスケールしない**（CON-002）

**回避策:**
- 信用を「判定」せず「伝達」する（パイプ）
- ルーターではなく「事業者側の標準ツール」を目指す
- 独占ではなく「デファクトスタンダード」（schema.org的立場）を狙う
- 信用は「結果の統計」で後付けで積む（引用率X%の実績）

**Sighted/AEOの現在地との整合:**
- 今やっていること（GPR実験、引用計測）は「結果で証明する」モデル → 正しい
- 「AIに正確に引用されましたか？」= 計測であって審判ではない → LAW-008回避
- 次のステップ: 計測→最適化→配信の一気通貫 = ツールとしての拡張
