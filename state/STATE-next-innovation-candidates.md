# イノベーション = ステップの消去

各世代のイノベーションは前の世代の「ステップを1つ以上消している」。速くしたのではなく、なくした。

| Before | After | 消えたステップ |
|---|---|---|
| カテゴリを辿る→見つける→読む | キーワード→見つける→読む | 辿る |
| キーワード→見つける→読む | 質問→答え | 見つける＋読む |
| 電話→待つ→乗る→行き先→支払い | タップ→待つ→乗る→着く | 電話＋行き先＋支払い |
| 店に行く→探す→選ぶ→買う | 検索→選ぶ→ワンクリック | 店＋探す |
| 辞書を引く→読む→理解する | AI翻訳 | 引く＋読む |

**次のイノベーション = 今のプラットフォームに残っているステップを消す**

# 経路A: 既存プラットフォームのステップ消去（PayPalモデル適合順）

## A-1: 買い物の「選ぶ」を消す

**現在**: Amazon/楽天で「検索→比較→選ぶ→買う」
**次**: 「必要になる→届く」— 選ぶステップが消える

| 項目 | 内容 |
|---|---|
| 消えるステップ | 検索＋比較＋選択 |
| どうやって | AIが過去の購買パターン＋消費速度から予測して自動補充/提案 |
| 因果即時か | ◎ 届いた/届かない、合っている/違う、は即座に分かる |
| 両面か | ◎ 消費者×商品のルーティング |
| ロックイン | ◎ 購買履歴＋好みデータが蓄積するほど精度が上がり抜けられない |
| 能力拡張か | ◎ 「考えなくても最適なものが来る」は新しい能力 |
| 窓 | △ Amazonが自社で実装可能（Subscribe & Save の発展系） |
| PayPal計 | 4.5 |
| **判定** | **★★ 有力だがAmazon/各社が自社実装する可能性高。独立プレイヤーの窓は狭い** |

実例の兆候: Amazon Dash（失敗したが方向は正しかった）、定期便、AIショッピングアシスタント

## A-2: 支払いの「支払う行為」を消す

**現在**: Apple Pay/PayPayで「タップ→認証→完了」
**次**: 「歩いて出る→自動精算」— 支払い行為が消える

| 項目 | 内容 |
|---|---|
| 消えるステップ | タップ＋認証（支払い行為そのもの） |
| どうやって | 顔認証/生体認証で入店、手に取って出るだけ（Amazon Go型） |
| 因果即時か | ◎ |
| 両面か | ◎ 消費者×店舗 |
| ロックイン | ○ 生体データ＋購買履歴 |
| 窓 | △ Amazon Go撤退。ハードウェアコストがボトルネック（LAW-007: 物理層） |
| PayPal計 | 4.0 |
| **判定** | **★ ハードウェアコストがまだ高い。CON-004通過だが物理制約** |

## A-3: 配車の「呼ぶ」「待つ」を消す

**現在**: Uberで「タップ→待つ(3-10分)→乗る→着く」
**次**: 「出たら目の前にいる→乗る→着く」

| 項目 | 内容 |
|---|---|
| 消えるステップ | 呼ぶ＋待つ |
| どうやって | AIが行動パターン/カレンダーから予測し、出発時刻に車が待機 |
| 因果即時か | ◎ 車がいる/いないは即座 |
| 両面か | ◎ 乗客×車両 |
| ロックイン | ◎ 行動データが蓄積するほど予測精度向上 |
| 窓 | △ Uberが自社実装可能。自律走行車の普及が前提 |
| PayPal計 | 4.5 |
| **判定** | **★★ 行動予測×自律走行の組み合わせ。ただしUber/Waymoが先行** |

## A-4: 質問の「質問を考える」を消す

**現在**: ChatGPTで「質問する→答えが出る」
**次**: 「知りたいことが出てくる」— 質問を考えるステップが消える

| 項目 | 内容 |
|---|---|
| 消えるステップ | 質問の言語化 |
| どうやって | AIが文脈（今読んでいるもの、今やっていること、今の課題）から「あなたが知るべきこと」をプッシュ |
| 因果即時か | ◎ 役に立った/立たなかったは即座 |
| 両面か | △ 利用者×情報源（ただし現在のAIモデルに吸収されやすい） |
| ロックイン | ◎ コンテキストの蓄積＝最大のロックイン |
| 能力拡張か | ◎ 「自分が知らないことを知らない」問題（WYSIATI）を解決する新能力 |
| 窓 | △ ChatGPT/Claude/Geminiが全社狙っている |
| PayPal計 | 4.5 |
| **判定** | **★★★ 最も革命的だが、既存AI各社が全力で狙っている。独立プレイヤーの勝ち筋は「特定ドメインの文脈理解」** |

## A-5: コミュニケーションの「打つ/話す」を消す

**現在**: LINE/WhatsAppで「打つ→送る→待つ→読む」
**次**: 「意図する→伝わる」— 入力ステップが消える

| 項目 | 内容 |
|---|---|
| 消えるステップ | タイピング/音声入力/文面作成 |
| どうやって | AIが文脈から「このメッセージを送るべき」を提案、承認だけ |
| 因果即時か | ◎ 送れた/送れないは即座 |
| 両面か | ◎ 送り手×受け手 |
| ロックイン | ◎ 会話パターン＋関係性データ |
| 窓 | △ Apple/Google/Metaが自社メッセンジャーに統合可能 |
| PayPal計 | 4.0 |
| **判定** | **★★ CON-008の壁: 人は「自分の言葉で」伝えたい（自律欲求）。AIが書いたメッセージは地位低下** |

## A-6: 予約の「探す→比較→予約する」を消す

**現在**: 食べログ/OpenTableで「探す→比較→電話/予約→行く」
**次**: 「行きたい→席がある」

| 項目 | 内容 |
|---|---|
| 消えるステップ | 探す＋比較＋予約手続き |
| どうやって | AIエージェントが好み×場所×空席を即座にマッチング、確認だけ |
| 因果即時か | ◎ 予約できた/できないは即座 |
| 両面か | ◎ 客×店のルーティング |
| ロックイン | ◎ 好みデータ蓄積 |
| 能力拡張か | ○ 「全レストランの空席が見える」は新能力 |
| 窓 | ○ 食べログ/Yelpは腐っている（LAW-008）。AIエージェント型は未確立 |
| PayPal計 | 4.5 |
| **判定** | **★★★ 食べログ型の腐敗した審判（LAW-008）を、AIエージェント型ルーティングで置換。因果即時。両面ネットワーク。好みデータでロックイン** |

## A-7: 学習の「教材を探す→順番に学ぶ→テストで確認」を消す

**現在**: Coursera/教科書で「教材選び→カリキュラム通り学ぶ→テスト→理解確認」
**次**: 「やりたいことをやる→足りない知識がその場で入る」

| 項目 | 内容 |
|---|---|
| 消えるステップ | 教材選び＋カリキュラム設計＋テスト |
| どうやって | 実際のタスクをやりながら、AIが「今ここで知るべきこと」を文脈内で提供 |
| 因果即時か | ◎ できた/できないは即座 |
| 両面か | △ 学習者×知識（ただしプラットフォーム化しにくい） |
| 能力拡張か | ◎ 「学ばずにできるようになる」は根本的な能力拡張 |
| 窓 | ○ Duolingo等が部分的。GitHub Copilotが「コーディングしながら学ぶ」を部分実現 |
| PayPal計 | 4.0 |
| **判定** | **★★ 特定スキル領域でのcopilot型。汎用化は難しい** |

# 経路B: 因果の短縮技術

## B-1: 保険/金融の「結果シミュレーション」

**現在の因果**: 保険が良かったかは事故発生時（年-一生）まで分からない
**因果短縮**: 「もし事故が起きたら、各社の保険でいくら出るか」を即座にシミュレーション

| 項目 | 内容 |
|---|---|
| 短縮前 | 事故→請求→支払い（年単位） |
| 短縮後 | シミュレーション結果が即座に表示（秒） |
| 信頼性 | △ シミュレーション精度に依存。過去の支払い実績データが鍵 |
| ロックイン | ○ シミュレーション履歴＋保険ポートフォリオ管理 |
| **判定** | **★★ 因果を「体験」から「シミュレーション」に変換。データがあれば成立** |

## B-2: 契約書の「判例確率表示」

**現在の因果**: この条項が不利だったかはトラブル時（年単位）まで分からない
**因果短縮**: 「この条項でトラブルになった確率X%、平均損害Y万円」を即座に表示

| 項目 | 内容 |
|---|---|
| 短縮前 | トラブル発生→訴訟→判決（年単位） |
| 短縮後 | 統計的リスクスコアが即座に表示（秒） |
| 信頼性 | △ 判例データの網羅性に依存 |
| **判定** | **★★ 因果を「体験」から「統計」に変換。法務データベースへのアクセスが鍵** |

# パターン分析

## 独立プレイヤーが取れる窓が開いているもの

A-4（文脈プッシュ）とA-6（予約の消去）が最有力だが、構造が異なる：

**A-4**: AI各社が全力で狙っている。独立プレイヤーの勝ち筋は**特定ドメインの深い文脈理解**（例：特定業界の営業マンが「次に何を知るべきか」を文脈からプッシュ）

**A-6**: 食べログ/Yelp型の腐った審判がAIエージェントに置換される。**飲食に限らず、あらゆる「予約/手配」に適用可能**（美容院、クリニック、修理業者、家事代行…）。各業界の予約システムはバラバラで統一されていない＝**ルーティングの空白**

## 経路Bで最も実現可能性が高いもの

**B-1（保険シミュレーション）**: データの入手が比較的容易（保険の約款は公開情報）。シミュレーション結果は即座に提供可能。因果を「体験」から「シミュレーション」に変換する最もクリーンな例。

## 共通する「ステップ消去」のパターン

消去されるのは常に**「人間が自分でやる認知的作業」**:
- 検索する → 消去
- 比較する → 消去
- 選ぶ → 消去
- 予約手続きをする → 消去
- 質問を考える → 消去
- 入力する → 消去

残るのは**「意図する」と「承認する」**だけ。意図→AI処理→承認→完了。この「意図→承認」の2ステップ構造が次世代のUIパターン。
