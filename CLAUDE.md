# CLAUDE.md — world-simulator 運用規約

このリポジトリは知識を溜める場所ではなく、予測を回す装置である。ポジショニングは**未知領域の探索装置**——法則（地図）と世界（領土）のズレを予測という賭けで計測する。AI セッションは以下の掟に従う。

## セッション開始時

1. `make validate` を実行し、期限切れの予測・腐敗したシグナル・STALE な現在地を検出する
2. 期限切れ（OVERDUE）の予測があれば、まず解決（resolve）してから新しい作業に入る
3. STALE な state/ があれば更新する（下記「現在地の更新プロトコル」）

## 時間の三層

情報は腐り方で置き場所が決まる：

| 層 | 場所 | 更新規則 |
|----|------|---------|
| 過去（不変） | `history/` | 追記のみ。確定した過去は腐らない。年表・コスト系列など法則と事例の原料 |
| 現在（定点） | `state/` | `refresh_every_days` 周期で上書き更新。周期超過で STALE 警告 |
| 変化（観測） | `signals/` | `expires` つき。予測に未接続なら腐って削除 |

## 現在地の更新プロトコル（state/）

state/ は「現在の世界の概要を安く手に入れる装置」である。更新手順：

1. 対象ドメインの世界線（simulations/）の「決定的観測」リストを読む。**観測は網羅ではなく、分岐確率を最も動かす順に行う**
2. WebSearch / スクレイピング / MCP で各観測項目の現状を取得する
3. state ファイルを上書きし、`as_of` を更新する
4. 前回からの差分のうち予測や分岐確率を動かすものは signals/ に切り出す（出典・信用クラスつき）
5. 決定的観測に該当が出ていたら、世界線の分岐確率を更新し、更新理由を世界線ファイルに追記する

state は使い捨てのスナップショットであり、履歴は git が持つ。ファイル内に旧情報を残さない。

## 新ドメイン参入時（blindspot pass）

energy / web3 / space 等へ拡張する最初のセッションでは、法則や予測を書く前に：

1. その領域の「自分が問えていない問い」を洗い出す（unknown unknowns の探索）
2. 問いを3つに仕分ける：予測に変換できる → predictions/ へ / 現状把握が必要 → state/ の観測項目へ / どちらでもない → 捨てる
3. `history/` にその領域の基礎年表・コスト系列を作る（法則適用の原料）
4. `state/STATE-<domain>.md` を作ってから、最初の世界線を書く

## 情報追加の掟

- **予測に接続しないシグナルは追加しない。** 追加時に `linked_predictions` を埋めるか、同時に新しい予測を作る
- シグナルには出典（`source` / `source_url`）と信用クラス（`source_class: S0-S3`）が必須
  - S0: 当事者の一次情報（自社実験、直接の取材・契約）
  - S1: 第三者の一次情報（公式発表、一次統計、決算資料）
  - S2: 二次情報（報道、業界レポート）
  - S3: AI生成・伝聞 — **原則入れない**。入れる場合は理由を本文に明記
- シグナルには `expires`（有効期限）が必須。期限切れで未接続のものは削除する

## 予測の掟

- 予測には `probability`（0-1）、`horizon`（期限）、`resolution_criteria`（第三者が判定できる解決基準）が必須
- 予測は必ず 1 つ以上の法則（`laws:`）を引用する。法則を引用しない予測は「勘」であり、登録できない
- 解決したら `predictions/open/` から `predictions/resolved/` へ移動し、`status: resolved`、`outcome`、`resolved_on`、`resolution_note` を記入して `make score` を実行する

## 法則の掟

- 法則には `kill_conditions`（この法則が死ぬ観測条件）が必須。書けないなら法則ではない
- 法則の `confidence` は感想ではなく scoreboard の成績を反映して更新する
- kill condition に該当する観測が出たら `status: falsified` にする。ファイルは消さない——死んだ法則も学習データである
- 新しい法則は必ず 1 つ以上の事例（cases/）を証拠として持つ

## 事例の掟

- 事例は「過去に世界を変えた出来事」を法則の証拠として構造化する
- 必ず「どの摩擦が下がったか」「資金はどこからどこへ流れたか」「この事例が法則に**反する**側面」を書く。都合の良い面だけの事例は禁止

## シミュレーションの掟

- 全シミュレーションに反世界（anti-thesis: 本線のテーゼが死ぬ世界線）の分岐を確率つきで含める
- 各分岐に先行指標（どのシグナルが観測されたらその分岐の確率を上げるか）を書く
- **決定的観測（Value of Information）のセクションが必須**：答えが分岐確率を最も動かす観測から順にランクづけする。「答えを知っても確率が動かない観測」はリストに入れない。state/ の更新はこのリストを巡回する

## 歴史の掟（history/）

- history/ は不変のメモリ。確定した過去の年表・データ系列・記録を置く
- cases/ との区別：cases/ は法則の**証拠として構造化された**事例（反する側面の記述必須）。history/ は構造化前の**原料**（年表・コスト曲線・数値系列）。cases/ や laws/ は history/ を参照してよい
- 追記のみ。過去は書き換えない。誤りが見つかったら訂正を追記する

## ドメイン拡張

現在のドメイン: `content`（発信・AEO・情報流通）。
エネルギー、web3、宇宙、EV などへ拡張する場合、新ディレクトリは作らない。`domains:` フィールドに追加するだけ。法則はドメイン横断で共有される——それがこのリポジトリの存在意義である。

## コマンド

```bash
make validate   # スキーマ・参照整合性・期限チェック（エラーで exit 1）
make score      # Brier採点 → scoreboard/SCOREBOARD.md 再生成
make test       # engine のユニットテスト
```
