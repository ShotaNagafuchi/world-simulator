# 主張

複数プラットフォームのデータを統合してユーザーに全体像を提供するサービスは、以下の3条件が**同時に**成立するため、構造的に失敗する：

1. **サプライ側が協力しない**: 統合はサプライ側のロックイン・情報優位・価格支配力を弱める。CON-009（インセンティブ超反応）により、サプライ側のインセンティブは「共有しない」方向に固定される
2. **ユーザーが金を払わない**: 統合の価値は慢性的・分散的（毎日少しずつ便利）であり、急性の痛みを解決しない。CON-013（双曲割引）により、分散的な将来価値は現在の支払い意思を生まない
3. **ビジネスモデルが罠**: ユーザーが払わないため、サプライ側から課金 → LAW-008（審判の腐敗）で中立性が崩壊。ユーザーの利益を代弁するはずのサービスがサプライの販売チャネルに転化する

# 成功した統合と失敗した統合の違い

| | 成功した統合 | 失敗した統合 |
|---|---|---|
| 例 | Google（Webページ統合）、Amazon（出品者統合）、Uber（ドライバー統合） | Mint（銀行口座統合）、HealthVault（医療記録統合）、保険比較サイト |
| サプライ側のインセンティブ | **統合されると得をする**（Webページ: アクセスが増える / 出品者: 売上が増える / ドライバー: 乗客が増える） | **統合されると損をする**（銀行: 比較されてロックイン弱化 / 病院: 患者流出 / 保険: 価格競争激化） |
| ユーザーの痛み | **急性**（今すぐ見つけたい、今すぐ乗りたい、今すぐ買いたい） | **慢性**（見えたら便利だけどなくても死なない） |
| 課金主体 | サプライ側が喜んで払う（広告、手数料） | サプライ側が抵抗する or 払うと腐敗する |

**鍵となる判定基準**: 統合される側（サプライ）が「統合されることで新しい顧客/売上を得る」なら成功。「統合されることで既存の優位性を失う」なら失敗。

# メカニズム

```
統合サービスを作りたい
  → サプライ側にAPIを要求
  → サプライ側の判断: 統合で得するか損するか？
    → 得する（顧客が増える）: 協力する → Google/Amazon/Uber型の成功
    → 損する（ロックインが弱まる）: 協力しない → API拒否 or 妨害
      → 規制で強制できるか？
        → できる（PSD2等）: 規制依存で成長 → institution層の速度に制約される
        → できない: サービスが成立しない

同時に:
  → ユーザーに課金できるか？
    → 急性の痛み（今すぐ必要）: 払う → PayPal型
    → 慢性の痛み（あると便利）: 払わない → Mint型（終了）
      → サプライ側から課金 → LAW-008で腐る → 比較サイト型の末路
```

# 痛みの急性 vs 慢性の判定

PayPalモデルが成立するには、痛みが**急性**（今すぐ解決しないと損する/困る）でなければならない。

| 急性の痛み | 慢性の痛み |
|---|---|
| 送金できない（PayPal） | 全口座の合計が見えない（Mint） |
| 情報が見つからない（Google） | 保険の重複が分からない |
| 車が来ない（Uber） | 医療記録が分散している |
| 言葉が通じない（翻訳） | サブスクの合計額が分からない |

急性: 「今この瞬間困っている」→ 支払い意思が高い
慢性: 「あった方がいいけど今は困っていない」→ 支払い意思がほぼゼロ

# 反証可能性

- 「サプライ側が損をする統合」が、規制なしに持続的に成功する事例を追跡する
- 慢性の痛みに対する有料サービスのチャーン率を記録する（年10%以下なら「払い続ける」の証拠）

# 予測への使い方

入力：(1) 統合対象のサプライ側が統合で得するか損するか、(2) ユーザーの痛みが急性か慢性か
出力：
- サプライが得する × 痛みが急性 = **成功する統合**（Google, Uber型）
- サプライが得する × 痛みが慢性 = △ 成長は遅いが成立可能
- サプライが損する × 痛みが急性 = △ 規制で突破可能
- サプライが損する × 痛みが慢性 = **構造的に失敗する統合**（Mint型）

**馬鹿げた摩擦リスト（STATE-absurd-frictions）の#1,#2がここに該当する。** 文脈の分断と情報の分断は「慢性の痛み × サプライが損する」の最悪の組み合わせ。PayPalモデル5.0で評価したのは誤り——急性/慢性の判定が欠落していた。
